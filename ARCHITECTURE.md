# System Architecture: RegisterKaro ITR Automation

The RegisterKaro ITR Automation platform is a three-tier system comprising a **UI Layer (Next.js)**, a **Service Layer (Node.js/Express)**, and an **Automation Engine (Playwright)**. It is designed to navigate complex external portals using an intelligent state-machine approach rather than static scripts.

## 1. State Model

Unlike traditional linear automation scripts (e.g., Selenium IDE recordings), this engine uses a dynamic State Machine. The entry point evaluates the current DOM of the page and categorizes it into an explicit `STATE`.

**Key States (`STATES` enum):**
- `UNKNOWN`: Default initialization or unrecognized page.
- `LOGIN_PAGE`: Entry portal for authentication/registration.
- `REG_BASIC_DETAILS`, `REG_CONTACT_DETAILS`, `REG_OTP`: Standard user registration flow.
- `FORGOT_PASSWORD_PAN`, `FORGOT_PASSWORD_METHOD`, `FORGOT_PASSWORD_OTP_CHOICE`: The recovery pivot path.
- `SET_PASSWORD`: Setting the final credentials.
- `SUCCESS` / `DASHBOARD`: Terminal completion states.

**Pivot Logic:**
If the bot begins registering a user and the portal surfaces an "Already Registered" error (or hits the `LOGIN_PASSWORD_PAGE` directly), the state evaluator automatically pivots to the `FORGOT_PASSWORD_PAN` state. This prevents brittle failures and adapts to existing user states dynamically.

## 2. Event Schema & Communication

Communication between the isolated Playwright bot and the backend is handled via Webhooks and a polling mechanism.

**Event Payload Schema (`emitEvent`):**
```json
{
  "jobId": "66782c5a0000000000000001",
  "seq": 1,
  "level": "info",    // 'info', 'warn', 'error'
  "phase": "INIT",    // State or Phase context
  "message": "Bot launched for PAN: ABCDE1234F"
}
```

The Bot POSTs these payloads to the Service layer's Webhook endpoint (`/webhook/events`).

## 3. Live Streaming & Replay

The Service layer acts as the event broker. When it receives a webhook from the bot, it pushes that event to connected clients (Next.js dashboard) in real-time.

1. **Live Streaming:** Uses Server-Sent Events (SSE) mapped to `text/event-stream`. Connected clients receive `<EventCard>` updates instantly as the bot navigates the portal.
2. **Replay:** Events are also pushed to MongoDB inside the `Job` document's `events` array. If a user refreshes the page or logs in later, the entire array is fetched over the standard API, seamlessly "replaying" the bot's historical actions chronologically.

## 4. Persistence Model

Data is stored in MongoDB using Mongoose. The primary entity is the `Job`.

**What is Persisted:**
- **Job Metadata:** `targetPan`, `registrationPayload`, `status`, `createdBy`.
- **Events Array:** The sequence of actions performed by the bot.
- **OTP Fields:** `suppliedOtp`, `lastOtpError`, `resendOtpRequested` (used for bidirectional sync).
- **Credentials:** `encryptedPassword`.

**Why:** 
- **Decoupling:** Persisting the OTP in the Job document allows the UI to simply `PATCH /jobs/:id` when the user enters an OTP. The bot independently polls the Job document (`GET /jobs/:id`) until an OTP appears.
- **Security:** The `encryptedPassword` generated by the bot is encrypted at the service layer using AES-256-CBC with a random Initialization Vector (IV) *before* touching the database.

## 5. Design Trade-offs

1. **State Machine vs. Linear Flow:** 
   - *Pro:* Highly resilient. Can handle dynamic popups, network timeouts, and logical pivots (like PAN already existing).
   - *Con:* Slightly higher CPU usage for continuous DOM evaluation and polling compared to simple `await page.click()`.
2. **Polling for OTP vs. WebSockets:**
   - *Pro:* The bot logic remains strictly decoupled. It doesn't need to maintain an active WebSocket connection with the backend. It simply makes a REST call every 2 seconds when blocked by an OTP gate.
   - *Con:* Introduces a 0-2 second latency between the user entering the OTP and the bot detecting it.
3. **Playwright Stealth:**
   - *Pro:* Bypasses standard bot-detection mechanisms present on government portals.
   - *Con:* Increases the container/dependency footprint (requires full Chromium binaries).
